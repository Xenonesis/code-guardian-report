/**\n * Python Taint Analyzer - Implementation Notes\n * \n * This document summarizes the design and architecture of the Python data-flow/taint\n * analysis component.\n \*/\n\n# Python Taint Analysis Engine\n\n## Overview\n\nThe Python Taint Analyzer (`src/services/analysis/PythonDataFlowAnalyzer.ts`) is an\nenterprise-grade data-flow/taint tracking engine designed to detect user-controlled\ndata flowing into dangerous operations in Python code. It uses **web-tree-sitter**\nwith the Python grammar to build an AST and then walks the tree to track variable\ntaint.\n\n## Key Design Decisions\n\n### 1. **Configuration-Driven Architecture**\n\nThe analyzer accepts a `PythonTaintConfig` object specifying:\n\n- **Sources**: entry points where tainted data originates (e.g., `input()`, `sys.argv`,\n Flask/Django request objects)\n- **Sinks**: dangerous operations where tainted data must not reach (e.g., `os.system`,\n `subprocess.run`, `eval`, `pickle.loads`)\n- **Sanitizers**: functions that clean data and break the taint chain (e.g., `escape()`,\n `urllib.parse.quote`, `html.escape`)\n\nThis design makes the analyzer extensible:\n\n`typescript\nconst customAnalyzer = new PythonDataFlowAnalyzer({\n  sources: [...defaultSources, \"my_custom_input\"],\n  sinks: [...defaultSinks, \"my_dangerous_call\"],\n  sanitizers: [...defaultSanitizers, \"my_custom_sanitizer\"],\n});\n`\n\n### 2. **Inter-Procedural Taint Tracking**\n\nThe analyzer tracks taint across function boundaries:\n\n1. Uses a `Map<string, string>` (`this.tainted`) to associate variable names with\n their originating source.\n2. When a tainted variable is passed as an argument to a user-defined function, the\n corresponding parameter inside that function is also marked tainted.\n3. When a function returns a tainted value and it's assigned to a variable, that\n variable becomes tainted.\n\n### 3. **Sanitizer Recognition**\n\nWhen the analyzer encounters a call to a known sanitizer (e.g., `escape()`,\n`sanitize()`, `urllib.parse.quote`), it:\n\n1. Checks if the call is a known sanitizer via `this.config.sanitizers`.\n2. If yes, *removes* taint from any variable assigned the result.\n3. Logs the sanitization for audit trails.\n\nExample:\n\n`python\nx = input()           # x is tainted\ny = sanitize(x)       # y is NOT tainted (sanitizer breaks the flow)\nos.system(y)          # No issue!  y has been cleaned\n`\n\n### 4. **Function Registration**\n\nBefore analyzing flows, the engine walks the AST once to register all\nuser-defined functions:\n\n1. Records function names and their AST nodes in `this.functions`.\n2. Extracts return identifiers (variables returned from the function) and stores\n them in `functionReturnVars` for simple return-taint propagation.\n\nThis allows later analysis to know which functions are user-defined and can be\nintra-procedurally analyzed.\n\n## Analysis Pipeline\n\n### Phase 1: AST Parsing\n\nThe analyzer uses `web-tree-sitter` to parse Python code:\n\n`typescript\nconst tree = this.parser.parse(content);\n`\n\n### Phase 2: Function Registration\n\n`typescript\nthis.walk(tree.rootNode, (node) => {\n  this.registerFunction(node);\n});\n`\n\nAll function definitions are extracted and stored.\n\n### Phase 3: Source, Propagation, Function Call, and Sink Detection\n\nIn a single tree walk, the analyzer:\n\n1. **Detects sources** (via `detectSource`): variables assigned from taint origins.\n2. **Detects propagation** (via `detectPropagation`): taint spreading through assignment\n and function return values.\n3. **Detects function calls** (via `detectFunctionCall`):\n - Checks if the call is to a user-defined function.\n - If any argument is tainted, marks the corresponding parameter tainted.\n - Recursively analyzes the function body for sinks.\n - Also checks for sanitizers and removes taint if encountered.\n4. **Detects sinks** (via `detectSink`):\n - Checks if a call is to a dangerous function.\n - If any argument is tainted, creates a `SecurityIssue` and logs it.\n\n## Supported Default Sources\n\n- `input()` – Python standard library user input\n- `sys.argv` – command-line arguments\n- `os.getenv()` – environment variables (can contain sensitive data)\n- `request.args`, `request.form` – Flask query/form parameters\n- `request.cookies`, `request.headers` – Flask HTTP metadata\n- `flask.request` – Flask request object\n- `django.http.HttpRequest` – Django request object\n- `werkzeug` – Werkzeug request handling\n\n## Supported Default Sinks\n\n- `os.system()` – OS command execution (Critical)\n- `eval()` – Python code evaluation (Critical)\n- `exec()` – Python code execution (Critical)\n- `subprocess.run()` – subprocess execution (High/Critical)\n- `pickle.loads()` – unsafe deserialization (High)\n- `yaml.load()` – unsafe YAML deserialization (High)\n- `requests.get()`, `requests.post()` – HTTP requests (Medium)\n- `urllib.request.*` – URL operations (Medium)\n\n## Supported Default Sanitizers\n\n- `urllib.parse.quote()` – URL encoding\n- `html.escape()` – HTML entity encoding\n- `escape()` – generic escape function\n- `bleach.clean()` – HTML sanitization\n- `re.sub()` – regex-based filtering\n- `sanitize()` – generic sanitizer convention\n\n## Integration with the Broader System\n\n### EnhancedAnalysisEngine\n\nThe main engine (`EnhancedAnalysisEngine`) automatically runs the Python analyzer\nas part of its analysis pipeline:\n\n`typescript\n// Phase 3b: Python-specific taint analysis\ntry {\n  await this.pythonAnalyzer.init();\n  const pythonIssues = this.pythonAnalyzer.analyzeDataFlow(fileContents);\n  allIssues = [...allIssues, ...pythonIssues];\n} catch (err) {\n  logger.warn(\"python taint analyzer failed\", err);\n}\n`\n\nThis is **opt-in via error handling**; if the Python analyzer fails, it does not\nblock the overall analysis.\n\n### EnhancedFileAnalysisService\n\nWhen analyzing individual files, the file service checks if the file is `.py` and\ninvokes the Python analyzer to append its findings.\n\n### MultiLanguageSecurityAnalyzer\n\nPython rules are *also* available as regex-based patterns in\n`MultiLanguageSecurityAnalyzer.ts`, providing an additional layer of pattern\nmatching for common issues (e.g., `os.system()` calls) that may not be caught\nby the taint engine.\n\n## Issue Format\n\nEvery taint flow detected by the analyzer produces a `SecurityIssue` with:\n\n- `id`: unique issue identifier\n- `type`: \"Python Data Flow\"\n- `category`: \"Injection\"\n- `message`: descriptive message (e.g., \"Tainted data reaches os.system\")\n- `severity`: \"Critical\" (command/code injection typically critical)\n- `confidence`: 95 (high confidence due to AST-based detection)\n- `filename`: source file\n- `line`: line number where sink is called\n- `column`: column number (0 for now; can be enhanced)\n- `codeSnippet`: empty (can be enhanced with source text)\n- `recommendation`: user-facing remediation advice\n- `remediation`: object with description, effort, priority\n- `riskRating`: \"Critical\"\n- `impact`: \"Remote command execution possible\" (varies by sink)\n- `likelihood`: \"High\"\n- `references`: OWASP/CWE links\n- `tags`: [\"python\"] + sink type\n- `tool`: \"Python Analyzer\"\n- `cvssScore`: 9 (CVSS v3.1 base score for command injection)\n- `cweId`: \"CWE-78\" (for command injection; varies by sink)\n- `owaspCategory`: \"A03\" (OWASP Top 10 2021 Injection category)\n\n## Testing\n\nComprehensive tests are provided in `src/tests/pythonAnalyzerIntegration.test.ts`\nand `src/tests/analysisAccuracyTest.ts` covering:\n\n- Source detection (input(), sys.argv, request.args, etc.)\n- Taint propagation through assignments, function calls, and returns\n- Sink detection (os.system, subprocess.run, pickle.loads, yaml.load, etc.)\n- Sanitizer recognition and taint removal\n- Issue metadata (CWE, OWASP, CVSS, recommendations)\n- Multiple file analysis\n- Edge cases (empty files, comments, nested calls)\n- Severity and confidence scoring\n\n## Limitations and Future Enhancements\n\n### Current Limitations\n\n1. **Single-Pass Analysis**: The analyzer walks the API tree once; it does not\n build a full control-flow or data-flow graph, so complex conditional/looping\n flows may be missed.\n2. **Limited Container Propagation**: Taint propagation through dictionaries,\n lists, and other containers is not yet implemented.\n3. **No Cross-Module Analysis**: The analyzer processes each file independently;\n it does not track taint across imported modules or resolve symbol imports.\n4. **Heuristic-Based Identifier Extraction**: The `findSourceInNode` and\n `extractIdentifiers` helper functions use string matching; they may miss\n complex expressions.\n5. **No Type Information**: The analyzer does not leverage Python type hints or\n type stubs to disambiguate names или validate flow semantics.\n\n### Recommended Future Enhancements\n\n1. **Build a CFG** for each function to properly model control dependencies.\n2. **Implement container/subscript taint propagation** so that tainted list\n elements and dict values are tracked.\n3. **Add cross-module analysis** by resolving imports and analyzing callers.\n4. **Leverage type hints** (PEP 484) when available to improve precision.\n5. **Add backward/reverse data-flow** to identify all consumer sto values that\n flow into a sink.\n6. **Implement context-sensitive analysis** so the same function called with\n both tainted and untainted data is analyzed separately.\n7. **Extend to more Python-specific sinks**: template engines, ORM raw queries,\n LDAP/SMTP operations, logging with format strings, etc.\n\n## Code Organization\n\n- paths and edge cases\n\n- **`test-python-parser.ts`** – quick smoke test\n - Can be run standalone: `npx tsx test-python-parser.ts`\n - Demonstrates basic functionality\n\n## Performance Characteristics\n\n- **Time Complexity**: O(n) per file, where n = number of AST nodes.\n- **Space Complexity**: O(m), where m = number of distinct tainted variables\n (typically small < 100 in most files).\n- **Typical Performance**: < 100ms per 1000 LOC Python file.\n- **Memo Efficiency**: One live `tainted` map per file (cleared between files).\n\n## Comparison with Static Analysis Tools\n\n|
